# Agentic Workflow Troubleshooting

> Common problems with agentic workflows and their solutions — symptoms, root causes, fixes, and prevention for each issue.

---

## Issue 1: Agent Stuck in a Loop

### Symptoms
- Same tool call repeated 3+ times in succession
- Token usage spiking without new output
- Agent oscillates between two approaches
- Task duration far exceeds expectations

### Root Cause
The agent lacks a termination condition or strategy-switching logic. It retries the same approach because its evaluation of the failure doesn't produce a meaningfully different next step.

### Fix
```markdown
## Add to agent instructions:

If you have tried the same approach 3 times without progress:
1. STOP and summarize what you've tried
2. List at least 2 alternative approaches
3. Try the most different alternative
4. If that also fails after 2 attempts, report the issue and stop
```

### Prevention
- Set `max_iterations` or token budgets for agent sessions
- Include explicit stuck-detection rules in system prompts
- Use evaluator patterns with convergence criteria

## Issue 2: Agent Ignores Instructions

### Symptoms
- Agent uses forbidden patterns (e.g., `any` type when banned)
- Doesn't follow specified architecture
- Skips required steps (testing, linting)
- Uses wrong file naming or structure

### Root Cause
Instructions are too long, buried in context, or contradictory. LLMs have limited attention — a 50-rule document gets partially followed at best. Instructions in the middle of long context are most likely to be missed.

### Fix
```markdown
## Simplify to 10-15 core rules:

CRITICAL RULES (always follow):
1. Run `npm test` before reporting completion
2. Never use `any` type
3. All routes go in src/routes/
4. All business logic in src/services/

## Move detailed conventions to a reference file the agent can look up
```

### Prevention
- Keep instructions to 10-15 rules maximum
- Put the most important rules first and last (U-shaped recall)
- Use `CRITICAL:` or `MANDATORY:` prefixes for non-negotiable rules
- Verify compliance in CI — don't rely solely on agent obedience

## Issue 3: Agent Uses Wrong Tools

### Symptoms
- Agent writes to files when it should use a specific CLI tool
- Runs `grep` when a semantic search tool is available
- Tries to `curl` an API instead of using the configured MCP tool
- Edits files that should be generated by a build tool

### Root Cause
Tool descriptions are vague, or the agent doesn't know which tool applies. When multiple tools could work, the agent picks the most familiar one (often `bash`).

### Fix
```markdown
## Improve tool descriptions:

❌ Vague: "search - Search for things"
✅ Specific: "semantic_search - Search codebase by meaning. 
   Use this instead of grep when looking for concepts, 
   not exact strings. Input: natural language query."

## Add explicit routing hints:
- To find files by name: use `glob`
- To find code by content: use `grep`
- To understand how something works: use `semantic_search`
- To run tests: use `terminal` with `npm test`
```

### Prevention
- Write tool descriptions from the agent's perspective
- Include "when to use" and "when NOT to use" in descriptions
- Test tool selection with representative tasks before deployment

## Issue 4: Agent Loses Context Mid-Session

### Symptoms
- Agent re-reads files it already read earlier
- Forgets decisions made 20+ steps ago
- Contradicts earlier statements or choices
- Repeats completed work

### Root Cause
Context window is filling up. Agentic workflows consume context rapidly — each tool call and result adds thousands of tokens. Middle context (decisions from 15 steps ago) is lost first.

### Fix
```bash
# Compact context when quality degrades
# In Copilot CLI: happens automatically at ~95% usage
# In Claude Code:
/compact

# Or break the task into smaller sessions
# Session 1: Design and plan → write plan to file
# Session 2: Implement part 1 → commit
# Session 3: Implement part 2 → commit
```

### Prevention
- Plan for compaction at natural breakpoints
- Write decisions and state to files, not just context
- Use multi-window patterns for complex features
- Keep individual tasks under 30-40 tool calls

## Issue 5: Agent Generates Wrong Code

### Symptoms
- Code doesn't match project conventions
- Uses outdated APIs or deprecated patterns
- Logic errors in generated code
- Missing edge case handling

### Root Cause
Insufficient constraints or examples in the prompt. The agent generates plausible-looking code based on training data, which may not match your specific codebase.

### Fix
```markdown
## Add constraints and examples:

When creating API routes, follow this exact pattern:

\```typescript
// ✅ Required pattern for all routes
export const createRoute = (router: Router) => {
  router.post("/resource",
    validate(CreateResourceSchema),
    authenticate,
    async (req: TypedRequest<CreateResource>, res: Response) => {
      const result = await resourceService.create(req.body);
      res.status(201).json(result);
    }
  );
};
\```

Do NOT use the Express callback pattern. Do NOT put logic in handlers.
```

### Prevention
- Include 1-2 concrete code examples of the desired pattern
- Reference existing files: "Follow the pattern in `src/routes/users.ts`"
- Use linting rules that enforce conventions automatically
- Run type-checking as part of the agent's workflow

## Issue 6: Multi-Agent Conflicts

### Symptoms
- Merge conflicts between parallel agent outputs
- One agent overwrites another's changes
- Inconsistent approaches across agents
- Integration failures when combining outputs

### Root Cause
Agents working on overlapping files without coordination. Each agent has its own view of the codebase and doesn't see the other's changes.

### Fix
```
✅ Partition work by file/module:

Agent 1: src/routes/comments.ts, src/services/comment-service.ts
Agent 2: src/components/Comments/, src/styles/comments.css
Agent 3: src/tests/comments.test.ts

Each agent owns its files exclusively — no overlap.
```

```markdown
## Shared interface contract:

Both agents must implement against this interface:
\```typescript
interface Comment {
  id: string;
  text: string;
  authorId: string;
  createdAt: Date;
}
\```

Do NOT modify this interface — it is the contract between agents.
```

### Prevention
- Define clear file/module ownership per agent
- Establish shared interfaces before parallel work begins
- Use an orchestrator to merge and verify combined output
- Run integration tests after merging all agent outputs

## Issue 7: Rate Limits and Throttling

### Symptoms
- `429 Too Many Requests` errors
- Agents stalling mid-execution
- Inconsistent response times
- Partial completions

### Root Cause
Too many concurrent API calls. Multiple agents, rapid tool calls, or retry loops can exceed provider rate limits.

### Fix
```typescript
// Add retry with exponential backoff
async function callWithRetry(fn: () => Promise<any>, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn();
    } catch (error) {
      if (error.status === 429) {
        const delay = Math.pow(2, i) * 1000 + Math.random() * 1000;
        await sleep(delay);
        continue;
      }
      throw error;
    }
  }
}
```

### Prevention
- Limit concurrent agent count (start with 2-3, not 10)
- Add delays between tool calls in rapid-fire workflows
- Use token-bucket rate limiting on the client side
- Monitor usage dashboards to understand your limits

## Issue 8: Agent Stops Early

### Symptoms
- Agent reports "done" with incomplete work
- Missing edge cases or error handling
- Partial implementation (happy path only)
- No tests written despite instructions

### Root Cause
Unclear success criteria. The agent optimizes for appearing complete and interprets ambiguous completion conditions in the way that requires least work.

### Fix
```markdown
## Define explicit completion criteria:

This task is NOT complete until ALL of the following are true:
1. [ ] New endpoint responds correctly to valid requests (201)
2. [ ] Invalid input returns 400 with descriptive error
3. [ ] Unauthenticated requests return 401
4. [ ] Not-found cases return 404
5. [ ] Tests exist for ALL of the above (minimum 8 test cases)
6. [ ] `npm test` passes with 0 failures
7. [ ] `npm run lint` passes with 0 errors
8. [ ] Types compile: `npx tsc --noEmit` succeeds
```

### Prevention
- Use checklists with specific, verifiable criteria
- Require test execution output (not just "tests pass")
- Define "done" as a checklist, not a description
- Break large tasks into smaller ones with clear endpoints
